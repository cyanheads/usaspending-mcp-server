# 0.2.9 ā€” 2026-07-17

Five fixes across the error surface and the tool descriptions: upstream misses and out-of-range dates now reach callers as declared, actionable errors rather than raw upstream failures, and `usaspending_spending_over_time` and `usaspending_get_federal_account` describe what the API actually returns.

## Added

- **`usaspending_search_awards` declares `date_before_earliest`** (`ValidationError`, non-retryable) ā€” a `start_date` before the new `EARLIEST_SEARCH_DATE` floor of `2007-10-01` is now rejected client-side with the boundary and a pointer to the Custom Award Download / `bulk_download` endpoints for data back to 2000-10-01. The endpoint answers such a request with a raw HTML 500, so the guard runs before the call. Covers both input paths ā€” the flat `time_period` and the nested `filters.time_period_start` resolve into the same start date. ([#38](https://github.com/cyanheads/usaspending-mcp-server/issues/38))

## Changed

- **`usaspending_spending_over_time` describes and renders `month` as a fiscal month** ā€” an ordinal within `fiscal_year` where 1 = October, not a calendar month. The tool description, the `group` input, and the output `time_period.month` field all say so; `format()` renders `FM1` instead of the unqualified `M1`, with a legend line emitted only when rows carry a month. Upstream returns one shape per `group` mode ā€” `{fiscal_year}`, `{fiscal_year, quarter}`, or `{fiscal_year, month}` ā€” now documented on `RawSpendingOverTimeResult.time_period`. ([#32](https://github.com/cyanheads/usaspending-mcp-server/issues/32))
- **`usaspending_get_federal_account` points callers at `usaspending_search_federal_accounts`** for account codes ā€” its tool description, `account_code` input, declared `account_not_found` recovery, and runtime recovery hint previously cited `usaspending_get_award`'s `account_obligations_by_defc`, which carries DEF codes and amounts, not `AGENCY-MAIN` account numbers. ([#20](https://github.com/cyanheads/usaspending-mcp-server/issues/20))
- **`usaspending_search_awards`'s `upstream_messages` describes the date advisory as boilerplate** ā€” the API returns its "search covers 2007-10-01 onward" note on every successful response regardless of the dates supplied, so it is not a verdict on the requested range. ([#38](https://github.com/cyanheads/usaspending-mcp-server/issues/38))

## Removed

- **`usaspending_spending_over_time`'s `time_period.calendar_year` output field and the `Cal Year` column** ā€” upstream only populates `calendar_year` under `group: "calendar_year"`, which this tool's `group` enum does not expose, so the field was unreachable and the column always rendered `N/A`. The handler's `tp.month ?? tp.calendar_month` fallback and both fields on `RawSpendingOverTimeResult.time_period` are gone with it; `calendar_month` appears in no live response under any mode. ([#32](https://github.com/cyanheads/usaspending-mcp-server/issues/32))
- **The unreachable `httpErrorFromResponse` guard in `USASpendingService.post<T>()`** and its now-unused import ā€” `fetchWithTimeout` throws a status-mapped `McpError` on every non-2xx before returning, so `response.ok` was always true at the guard. Behavior-preserving: POST-path failures surfaced then and now as the framework's status-mapped error. ([#40](https://github.com/cyanheads/usaspending-mcp-server/issues/40))

## Fixed

- **Upstream 4xx misses no longer bypass the declared not-found contracts** on `usaspending_get_award`, `usaspending_get_recipient`, `usaspending_get_agency`, and `usaspending_get_federal_account` ā€” `fetchWithTimeout` threw before each handler's existence check could run, so a miss surfaced as an unclassified fetch error with no `data.reason` and no recovery hint. A private `getEntity<T>()` helper now resolves `undefined` on the statuses that mean "no such entity" at these four endpoints (400 and 404, verified live per endpoint), letting each handler's `ctx.fail()` fire. The shared `get<T>()` primitive is untouched, so `listAgencies`, `getAgencySubAgencies`, `getAgencyBudgetaryResources`, and `getDisasterOverview` still surface their 4xx as real errors instead of empty successes; non-miss statuses ā€” including the 422 the agency endpoint raises for an out-of-range `fiscal_year` ā€” still propagate. ([#27](https://github.com/cyanheads/usaspending-mcp-server/issues/27))
